# M10 — Casting/Streaming Implementation Plan (second half of M10)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Complete M10 — let participants surface their streams (embed/link) and light up the presence "who's streaming" facet, give casters **auto-updating OBS browser-source overlays** (bracket + scoreboard) that reuse the existing M3 bracket UI / M6 CS2 scoreboard and update live over Reverb, and add small per-game spectator recipes. After this, M10 gets its full `m10` tag.

**Architecture:** Streams are a `users.stream_url` field (participant self-service in the profile page) surfaced on the existing `ParticipantPresence` DTO + a "nur Streams" filter on the presence page. Overlays are **public, no-auth, transparent-background Inertia pages** under `pages/Overlay/` (layout `null`, mirroring the `/screen/{event}` beamer pattern) that reuse `BracketView.vue` (render-only) and `SceneScoreboard.vue`, driven live over the existing public `tournament.{id}` channel. Spectator recipes copy the `InstallHint`/`InstallHintCast` pattern verbatim as `SpectateHint`/`SpectateCast` on the `games` catalog. No new external systems; no new broadcast infrastructure (reuse `.match.*` events).

**Tech Stack:** PHP 8.4 · Laravel 13 · Filament v5 · Inertia v2 + Vue 3 + Tailwind v4 + shadcn-vue · Reverb · Pest 4.

**Scope:** IN — `users.stream_url` + profile field + presence stream facet/filter; bracket overlay; scoreboard overlay; per-game spectator recipes. OUT (deferred) — the **friends** filter (still needs M9 friends) and context-sensitive names (M9); a full live-player *embed* inside the presence board (we LINK; a dedicated embed view is a possible later enhancement); any stream-liveness polling (we do not call Twitch/YouTube APIs).

## Global Constraints

- Code, comments, commits, docs in **English**; all UI copy in German via `lang/de/` (extend `lang/de/presence.php`, `lang/de/games.php`; add `lang/de/overlay.php`).
- **Conventional Commits** (`feat(presence): …`, `feat(overlay): …`, `feat(games): …`). TDD: failing test first wherever there is testable behavior; frequent commits; `composer check` (pint --test, phpstan level 8, pest — SEQUENTIAL, `-d memory_limit=1G`) + the four frontend gates (`npm run lint:check`, `format:check`, `types:check`, `build`) green after **every** task.
- **Modular monolith:** overlay HTTP lives in a new `app/Modules/Casting/Http/` (or reuses Tournaments/Infoscreen where a projection already exists — prefer reuse); Presence/Games changes stay in their modules. Read cross-module data through existing models/projections (`BracketMatchProjection`, `GameMatch`, `Tournament`, `User`), never raw foreign-table queries.
- **`Event` is the aggregate root.** Every overlay is gated by the owning event's **public visibility** (`Event::isPubliclyVisible()` → 404), exactly like `/screen/{event}` and every public participant page. Overlays are **no-auth** (an OBS browser source cannot log in) and therefore may expose **only already-public data** (bracket standings + live scores are already public on the beamer/tournament page). No private data on any overlay or the `tournament.{id}` channel. Privilege/state fields never `$fillable`; `stream_url` is a benign user preference (fillable is correct).
- **Signalpult design system is binding** (`docs/design.md`): semantic role tokens only (no raw hex), JetBrains Mono for machine data (scores, times), rationed amber, `LiveIndicator` for live state, four states, calm participant page / **loud, legible-from-across-the-room overlays**, Rams quality floor (responsive, visible focus, `prefers-reduced-motion`, AA contrast). Overlays additionally must have a **transparent background** so OBS composites them over gameplay. **Invoke the `frontend-design` skill first** for every UI task (10.5 presence facet, 10.6 + 10.7 overlays, 10.8 spectator display).

## Design decisions (locked for this phase)

- **Stream URL:** a single nullable `users.stream_url` (permissive `url` validation, any https — we don't restrict to Twitch/YouTube). Surfaced on the presence board as a **link** with a stream glyph/label (NOT an embedded live player — embedding needs a parent-domain handshake and is out of the base scope). The presence **"nur Streams" filter** shows only participants who have a `stream_url`. This completes the streams facet the M10-base plan explicitly deferred.
- **Overlay routes (public, no-auth, transparent):** `/overlay/tournament/{tournament}/bracket` and `/overlay/match/{match}/scoreboard`. Bracket is per-tournament; scoreboard is per-match (a caster casts one match at a time). Each `abort_unless(<owning event>->isPubliclyVisible(), 404)`. Pages under `resources/js/pages/Overlay/` → layout `null` (add the `Overlay/` prefix to `app.ts`'s layout switch). Wrap in a transparent full-viewport container (a transparent variant of `SceneFrame` — NOT `bg-background`).
- **Overlay liveness:** the **bracket** overlay subscribes with `useTournamentChannel`-style logic and does `router.reload({ only: ['matches'] })` on `.match.*` events (bracket data is persisted → reload is correct). The **scoreboard** overlay subscribes to `tournament.{id}` for `.match.score_updated`, filters to its `match_id`, and updates **reactive local state directly from the broadcast payload** (because the CS2 `round` is only in the event payload, not persisted on the match) — mirroring how `BroadcastScoreboardOnScoreUpdated` feeds the beamer `SceneScoreboard`. Initial render seeds from the match's persisted `team1/team2/score1/score2` (round omitted/0 until the first live tick).
- **Spectator recipe:** `SpectateHint{gotvConnect?: string, observerNote?: string, replayNote?: string}` — a jsonb `games.spectate_hint`, copying `InstallHint`/`InstallHintCast`/the Filament flat-field pattern verbatim. Displayed on the participant match/server surface as "So schaust du zu" when non-empty.
