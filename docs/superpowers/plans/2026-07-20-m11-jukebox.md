# M11 — LAN-Radio / Jukebox Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** A per-event communal hall playlist that checked-in participants steer by up-voting: search tracks, enqueue them, the vote order drives what plays next, mirrored into a Music Assistant player; now-playing shown on the beamer.

**Architecture:** LANoMAT owns the vote-ordered queue (source of truth) and is the voting/remote-control layer; **Music Assistant** (own LAN server, Docker) is the player/provider/queue engine behind a swappable **`MusicClient`** contract (MA = first implementation). LANoMAT never touches audio bytes — it commands MA over its HTTP/JSON-RPC API (Bearer token) and **polls** MA for now-playing, broadcasting changes over the existing `event.{id}` Reverb channel. **Execution = Mode A:** contract + `HttpMusicClient` (built against MA's documented API, `Http::fake`-tested) + `FakeMusicClient`; MA itself as Docker-compose config + setup docs; real-MA verification deferred (like M6-T13/M7/M8). If MA is down, only the jukebox pauses — no core-feature outage.

**Tech Stack:** PHP 8.4 · Laravel 13 · Laravel HTTP client (`Http::withToken`; `Http::fake` in tests — NEVER a real MA call) · Reverb · Inertia v2 + Vue 3 + Tailwind v4 + shadcn-vue · Pest 4 · PostgreSQL 16 · Redis.

## Global Constraints

- Code / comments / commits / docs in **English**; all UI copy in German via `lang/de/jukebox.php`.
- **External systems only through the `MusicClient` contract + `FakeMusicClient`** — feature/domain tests use `fakeMusic()`; the real `HttpMusicClient` is unit-tested only with `Http::fake` (never a real Music Assistant call). The exact MA REST wire format (JSON-RPC envelope, command paths) is confirmed against a running server's `:8095/api-docs` at real-infra time — that verification is the deferred Mode-A part; the contract + LANoMAT-side behavior are fully built and tested now.
- **Every authorization goes through a Policy.** Acting user is always `auth()->user()`; never a client-supplied id.
- **Settled product decisions (binding):**
  - **Participation = checked-in participants only** — a user may queue/vote only for an event they are checked in to (`EventRegistration.checked_in_at !== null`). Plus an **anti-flood cap: at most one still-unplayed queued item per user per event**.
  - **Vote-ordering = up-votes only** — the upcoming queue is ordered by net up-vote count desc (tiebreak: earlier `created_at` first); the most-voted unplayed item plays next. No down-votes.
  - **Skip = community threshold + orga override** — a configurable skip-vote threshold (default: a majority of currently checked-in participants, min 3) skips the current track; orga/helper may skip or remove any item immediately.
  - **"Discord verstärkt, ersetzt nie"** — the jukebox is in-app; no Discord control surface required.
- `MusicClient` capability segregation: core verbs `search`/`syncQueue`/`nowPlaying`/`skip` are shared by ALL backends; auth/device/setup live per-backend OUTSIDE the contract (config-bound); playback/device control (`pause`/`resume`) is an OPTIONAL capability interface implemented only by backends that support it — NOT a fat interface with no-op/`NotSupportedException`.
- **Conventional Commits** (`feat(jukebox): …`); TDD. `composer check` (pint --test, phpstan **level 8**, pest **SEQUENTIAL** — never `--parallel`; on `SQLSTATE[40P01]`/"table does not exist" check `ps aux | grep vendor/bin/pest` for strays) green after every task. Frontend gates green for any Vue change. New test files use `uses(RefreshDatabase::class)`.
- **Design system binding** (`docs/design.md` "Signalpult"): semantic tokens (no raw hex), rationed amber, four states, calm app / loud beamer, `LiveIndicator` for the live now-playing state, `font-mono` only for machine data (durations/counts). **Invoke `frontend-design` before Task 7 (participant UI) and Task 8 (beamer scene).**
- Reverb `event.{id}` is PUBLIC — the now-playing/queue broadcast carries only already-public track metadata (title/artist/art/vote counts), never user PII beyond public display names.

### Task 4: Queue/vote/skip actions + policy + ordered read-model

**Files:**

- Test: `tests/Feature/Jukebox/QueueAndVoteTest.php`, `tests/Feature/Jukebox/SkipTest.php`, `tests/Unit/Jukebox/JukeboxQueueOrderingTest.php`

**Interfaces:**
- `JukeboxPolicy`: `participate(User, Event)` = the user has an `EventRegistration` for the event with `checked_in_at !== null`; `moderate(User, Event)` = `isHelper()` (orga/helper override). Actions authorize via the policy.
- `AddToQueue::handle(User, Event, TrackDto): JukeboxItem` — guards (throw `JukeboxException`): must `participate`; **anti-flood** — reject if the user already has a `Queued|Playing` item in this event. Creates a `Queued` item from the `TrackDto`.
- `ToggleVote::handle(User, JukeboxItem): void` — must `participate` in the item's event; toggles the unique up-vote row (add if absent, remove if present).
- `ToggleSkipVote::handle(User, JukeboxItem): void` — must `participate`; toggles a skip-vote; after toggling, if skip-votes ≥ `SkipThreshold::for($event)` and the item is `Playing`, mark it `Skipped` (and let Task 5's sync advance MA).
- `SkipCurrent::handle(User, Event): void` + `RemoveItem::handle(User, JukeboxItem): void` — require `moderate`; orga override skip/remove.
- `SkipThreshold::for(Event): int` — `max(3, ceil(checkedInCount / 2))` (configurable via `config('jukebox.skip_ratio')`, default 0.5).
- `JukeboxQueue::upcoming(Event): Collection<JukeboxItem>` — `Queued` items ordered by `voteCount desc, created_at asc`; `current(Event): ?JukeboxItem` — the `Playing` item. Pure read-model (eager-load votes; no N+1).
